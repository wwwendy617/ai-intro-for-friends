亲测有效的 AI 协作原则
  • 先拆自己的活，再找工具
    不是"AI 能干什么"，而是"我的工作里有哪些环节"。先拆清楚业务流程，才知道 AI 该插在哪一步。工具不重要，工作流才重要。
  • 永远多问一个问题
    AI 的第一个答案是"平均水平答案"。追问一句——"有没有什么坑""换成我的场景要怎么改"——从第二个问题开始才进入真正有用的那一层。
  • 别怕弄坏——你弄不坏
    AI 对话随时开新的，代码随时回滚，文件改错了可以撤销。试错成本几乎是零。最大的坑不是"搞砸了"，是"不敢开始"。
  • 看不懂的就截图问 AI，别害羞，直接问
    报错、英文文档、看不懂的表格——截图发给豆包 / ChatGPT / DeepSeek，问"这张图在说什么"。别怕问得蠢——AI 不会笑话你，十次里有八次能直接解决。
  • 本地文件的活，交给 Cursor
    Cursor 不只是写代码——它是能操作你本地文件的 Agent。批量改文件、整理文件夹、读一堆文档做总结、跑脚本——把整个活儿丢给它，比自己一个个操作快十倍。

AI 世界地图

五层结构——看到任何新词都能找到它在哪一层

一个底层认知

现在几乎所有 AI 产品，本质上都是同一个东西——大模型——披着不同的外衣站在你面前。

外衣有五层，从最里面的核心到最外面的产品，长这样：

L5
垂直工具
妙鸭 / Midjourney / Cursor / Notion AI / Gamma…
L4
Agent 代理人 / 智能体
能自己拆任务、调工具、写记忆 — OpenClaw 在这一层
L3
Workflow 流程编排
人画好流程图，模型按图走 — Coze / Dify / 扣子
L2
Chatbot 对话机器人
大模型 + 聊天外壳 — 豆包 / ChatGPT / DeepSeek 网页版
L1
大模型 LLM
一个会接龙的超大文件 — GPT / Claude / DeepSeek / 豆包后端

每一层都把内层"包"起来，再加一点自己的东西。英文术语保留是故意的——下次你在文章、播客、产品介绍里撞见 "Agent"、"Workflow"、"Chatbot"，看一眼就知道它在哪一层。每一层我都给两件事：它是什么 + 它为什么能这样。

第一层：大模型（LLM）

它是什么

大模型就是最里面那一层。所有你听过的 ChatGPT、豆包、DeepSeek、通义、Claude、Gemini——它们的"脑子"都是大模型。

大模型本身不是产品，它就是一个文件。一个非常大的文件，里面装的是几千亿个数字（业内叫"参数"）。这个文件本身不会说话，也不会聊天，也不会画画。它只能干一件事：

你给它一段话，它预测下一个最可能出现的字是什么。

就这么简单。一个字一个字接龙接出来的。

它为什么能这样

这就是大家最容易卡住的地方——"就接龙吗？那它怎么会写代码、会讲笑话、会做 PPT 大纲？"

它能这样，是因为它接的不是普通的龙，是在几乎全人类的公开文本里训练过的接龙。

简单说三件事：

  1. 预训练：它把互联网上能拿到的几乎所有文本——书、论文、新闻、代码、维基、问答——全部读了一遍。读的过程不是"理解"，是反复练习"猜下一个字"。猜对就奖励，猜错就调整那几千亿个数字里的一部分。这一步它学会的是世界的统计规律：什么样的话后面接什么样的话。
  2. 对齐训练：光会接龙还不行，会接成一个胡言乱语的接龙怪。所以训练的人雇了一批人，给它示范"什么是好的回答 / 什么是坏的回答"，再用人的反馈微调它的行为。这一步让它听得懂人话，知道"你问问题的时候它要回答你"，而不是接着你的话往下编。
  3. 越大越聪明：参数从几亿涨到几千亿之后，会出现一个反直觉的现象——它自己长出没人教过它的能力，比如一步一步推理、做数学题、写代码。这个现象业内叫"涌现"，目前没有完美的解释，但所有大模型公司都在朝同一个方向赌：堆更多数据、更多算力、更大的模型。

一句话总结：

大模型 = 一个被全人类文本喂大的、会接龙的超大文件。

它本身不联网、不记得你、不会主动干活。就是一个文件。

第二层：Chatbot（对话机器人）

它是什么

豆包、ChatGPT、DeepSeek 网页版——这些都是 Chatbot。中文叫"对话机器人"。下次你在任何文章里看到 "chatbot" 这个词，知道它说的就是这一类产品。

Chatbot = 大模型 + 一个聊天界面 + 一点点把你说的话喂给模型的逻辑。

你打字，它把你说的话送进大模型，大模型一个字一个字接龙出回答，再把字一个一个推到屏幕上。

就这样。

它为什么能"记得"你前面说了什么

这是一个特别值得讲清楚的点。因为大模型本身没有记忆。它每一次回答都是从零开始的。

那 Chatbot 为什么能接住你前面说过的话？

因为 Chatbot 这一层做了一个很笨但很有效的事：每次你发送消息的时候，它把你们前面所有的对话历史，连同你这次的新问题，一起重新塞给大模型。

大模型每次都"读完前情提要再回答"。它不是记得你，是每次都被重新告知一遍你是谁、你前面说了什么。

这也解释了几件事：

  • 为什么聊太久会越来越慢、越来越不准——前情提要太长，模型读不动了。
  • 为什么开新对话它就完全不认识你了——历史没塞给它。
  • 为什么有些产品有"记忆"功能——是产品端帮你存了一份摘要，每次悄悄塞进前情提要里。

Chatbot 就是给大模型套了一个"对话外壳"，让它看起来像在和你聊天。

第三层：Workflow（流程编排）

它是什么

Workflow 中文叫"流程编排"。意思是：把大模型当成流水线上的一个工人，让它固定做某一步活。

比如客服自动回复的 Workflow：

用户消息 → 关键词识别 → 调用模型生成回复 → 发回给用户

或者文案自动生成的 Workflow：

读 brief → 调模型生成五个标题 → 调模型给每个标题写一段正文 → 输出表格

每一步该干什么，是人提前画好的。模型只是流程图里的一个节点。

国内你可能听过的 Coze、Dify、字节扣子、阿里百炼，还有飞书的智能伙伴——这些大体上都属于 Workflow 这一层。

它为什么能跑起来

它能跑起来不是因为模型变聪明了，是因为人把流程拆得足够细，每一步只让模型干一个它干得动的小活。

这也是 Workflow 的优点和缺点：

  • 优点：稳定、可控、好排错。每一步在哪儿断了都能查得出来。
  • 缺点：它不会自己思考。流程图画错了、走到一个画图的人没想到的分支，它就傻在那儿了。

你可以把 Workflow 理解成：一个预先设计好动线的工厂。模型是工人，流程图是动线。你换一个新订单，工厂未必能干。

第四层：Agent（能自己干活的代理人）

它是什么

这一层是最近一两年 AI 行业真正的主战场。也是 OpenClaw 所在的那一层。

Agent 中文一般翻译成"代理人"或"智能体"。简单粗暴的定义就是：

Agent = 大模型 + 一双手脚（能调用 Tool）+ 一个循环（能自己决定下一步干什么）+ 一段记忆

Agent 跟 Workflow 最大的差别是：

  • Workflow 是人画好图，模型按图走。
  • Agent 是给模型一个目标，让它自己决定要走哪几步。

举个对比：

Workflow Agent
输入 一个固定的输入格式 一个目标，比如"帮我把这周的微博舆情整理成一份给客户的简报"
步骤 人提前画好 AI 自己边想边决定
工具 在流程图节点里固定调 AI 看情况决定要不要调、调哪个
出问题 流程断在哪一步 AI 自己尝试换一个方法

它为什么能自己干活

这是今天最值得停下来讲清楚的一段。

它能"自己干活"，靠的是三件事的拼装：

第一件：工具调用（手脚）

光有大模型，它只会输出文字。但人们发现一件事——你可以让大模型输出一段"调用指令"，比如"我想用浏览器打开这个网址"，然后外面的程序真的去帮它打开浏览器，再把结果塞回前情提要给模型看。

这一下，模型就有手有脚了。它能上网、能读写文件、能发消息、能跑代码。

第二件：观察—思考—行动—再观察的循环

模型每次行动完，会被告知"刚才那个动作的结果是什么"。然后它再决定下一步。这个过程长这样：

1
思考
下一步该做什么
→
2
行动
调用一个 Tool
→
3
观察
看看结果是什么
↻ 回到第 1 步，直到目标达成才停下来

举个例子，你给它一个目标"帮我查一下半固态电池行业现在头部三家是谁"：

  1. 思考：我得搜一下"半固态电池 头部企业"。
  2. 行动：调用浏览器，搜这个关键词。
  3. 观察：搜出来一堆链接，前几条是新闻和行研报告。
  4. 思考：行研报告更靠谱，我点开第二条。
  5. 行动：调用浏览器读取第二条的正文。
  6. 观察：里面提到了五家公司。
  7. 思考：但我需要的是"头部三家"，再交叉验证一下其它来源。
  8. ……

这个循环它会自己跑十几步、几十步，直到它觉得问题答完了，才停下来给你回答。

第三件：磁盘上的 markdown 文件（记忆）

这是今天最反教科书、但也最值得你记住的一段。

市面上讲 Agent 的文章一般会告诉你："Agent 有短期记忆和长期记忆。" 这句话不能算错，但没有解释任何事。

真实情况是这样的：

Agent 的"记忆"，本质上就是存在你电脑里的一堆 markdown 文件。

模型本身没有记忆——它就是第一层里那个"会接龙的超大文件"，每次被调用都是从零开始的。它之所以"记得你"，是因为外面那层程序每次跟它说话之前，都会自动打开几个 markdown 文件、把里面的内容塞进前情提要给它看。

OpenClaw 官网的原话说得最直接：

"The model only 'remembers' what gets saved to disk — there is no hidden state."

模型只记得那些被写进磁盘的东西——没有任何隐藏的记忆。

这些文件就躺在 ~/.openclaw/workspace/ 下面，一共就这么几个：

  • MEMORY.md——长期记忆。你告诉过它的那些"持久"的事——偏好、决定、硬事实。每次一开对话就被自动读进去。
  • memory/2026-04-10.md（每天一份）——当天的日志。今天和昨天这两份会被自动读进来，作为近期的上下文。
  • DREAMS.md（实验中）——"做梦"机制写下来的沉淀。OpenClaw 会定期"做梦"：把日志里值得长期记住的东西自动提炼写进 MEMORY.md，把没价值的丢掉。这让长期记忆不会越滚越肥。

"做梦"听着玄，其实是一个非常聪明的工程设计——让 AI 自己定期整理自己的记忆，从"今天发生了什么"里沉淀出"这件事值得我长期记住"。

这才是这一段的关键：

Agent 的"记忆"——不在某家 SaaS 公司的服务器里，不是黑盒，也不是什么神秘的"向量数据库"。就是你自己电脑里的几个 markdown 文件。打开能读，不爽能改，备份就是复制，删除就是扔进回收站。

这也是 OpenClaw 这类 Agent 最反 SaaS 的地方——你跟它说过的所有话、它学过的所有东西、它的"成长轨迹"，全部在你自己的工作区里，看得见、改得动、搬得走。

三件事拼起来

有了这三件事——会用的手脚、能自己跑的循环、写在磁盘上的 markdown 记忆——模型就从"会接龙的文件"变成了一个能自己干活的代理人：

手脚
Tool
+
循环
Loop
+
记忆
Memory (md)
=
Agent
代理人

这一层目前能干到什么程度

我自己用 OpenClaw 跑过的真实任务里，它已经能：

  • 自己上网搜资料、读完十几篇内容、整理成一份带引用的简报
  • 自己读一份很长的 PDF，按我给的框架抽信息进 markdown 表格
  • 自己拆我们公司一个真实 brief，按我教过它的方法给出第一版策略草案
  • 自己审一篇内容稿，对照我设定的红线挑问题

这些事情没有任何一步是我手动操作的。我只是给它一个目标，它自己跑。

但它也不是万能的。它会犯错、会绕远、有时候会卡住。这也是为什么需要"教养"——后面会展开讲怎么教它。

第五层：垂直工具

它是什么

垂直工具就是把上面四层任何一层，包装成一个面向特定场景的产品。

营销人最熟的：

  • 妙鸭、Midjourney、即梦——图像生成
  • Runway、Sora、可灵——视频生成
  • Notion AI、Gamma、Tome——文档/PPT
  • Cursor、Windsurf、Trae——写代码
  • 11Labs、海螺——语音克隆

这一层不需要你理解里面的技术，你只需要知道一件事：

它们好不好用，主要看两件事——里面用的大模型够不够强、外面套的产品交互够不够贴你的场景。

里层弱，外层再贴心也没用；里层强，外层不贴你的场景，你也用不顺手。

术语速查

下次在文章、播客、朋友圈里撞见这些词，不用从零开始懵——你已经知道它在说什么了。

术语 人话翻译 在哪一层
LLM Large Language Model，大语言模型——就是那个会接龙的超大文件 L1
参数 / Parameters 模型文件里的数字，几千亿个。参数越多，模型越大，通常越聪明 L1
Token 模型读写的最小单位。一个中文字 ≈ 1-2 个 token，一个英文单词 ≈ 1 个 token L1
Context Window 上下文窗口——模型一次能"看到"多少内容。窗口越大，能读的前情提要越长 L1-L2
Prompt 你发给 AI 的那段话。写好 prompt = 把需求讲清楚 L2
System Prompt 系统提示词——产品提前塞给模型的"人设"，你通常看不到 L2-L3
RAG Retrieval-Augmented Generation——先搜资料，再让模型基于搜到的内容回答。知识库类产品的核心原理 L3
Hallucination 幻觉——AI 一本正经地胡说八道。接龙接到没见过的地方就会编 L1-L2
Fine-tuning 微调——拿你自己的数据再训练一遍模型，让它更懂你的场景。成本高，一般人用不到 L1
Tool Use / Function Calling 工具调用——让模型输出"我想调这个工具"，外面的程序真的帮它调。Agent 的手脚 L4
MCP Model Context Protocol——让 AI 连接外部工具的标准协议。类似"AI 世界的 USB 接口" L4
Vibe Coding 氛围编程——不写代码，用嘴告诉 AI 你想要什么，它帮你写 L4-L5
开源 / Open Source 代码公开，谁都能看、能改、能用。意味着你的数据和成果归你自己 通用
API Application Programming Interface——程序之间对话的接口。你用的产品背后都在调 API 通用

上手 OpenClaw

从认识到装好到教养

A. OpenClaw 是什么？为什么是它？

它是什么

OpenClaw 就是一只 Agent——第四层讲的那种"能自己干活的代理人"。

它的作者是一个国外的开发者 Peter Steinberger，他之前做过一个叫 PSPDFKit 的产品。OpenClaw 是开源的，吉祥物是一只叫 Molty 的太空小龙虾。"Claw"就是"爪"，这也是为什么我管它叫"虾"——它确实是一只虾。

它的官方 slogan 是：

"The AI that actually does things."

（那只真的会干活的 AI。）

它的内部结构（只要知道四层）

OpenClaw 内部有四层结构。不需要记技术名词，但知道这四层，后面"教养它"才有抓手：

01
Gateway 网关层
前台接待 — 接你的指令、把它路由到合适的脑子和手脚
02
Brain 大脑
接进来的大模型本身 — Claude / GPT / DeepSeek / Kimi / Qwen 任你换
03
Tools 手脚
一组它能调用的具体能力 — 执行命令、读写文件、上网、发消息、跑代码
04
Memory & Workspace 记忆与工作区
长期记忆、Skill 文件、日志 — 全是 markdown，存在你自己的电脑上

四层拼起来，就是一只能自己思考、自己动手、自己记住的 AI。其中第二层 Brain 不绑死任何一家——你可以随时换更聪明的脑子，这是 OpenClaw 最大的优点之一。

为什么是它（而不是别的）

我用过国内国外好几个 Agent 类产品，最后落在 OpenClaw 上，因为它在三件事上都做到了：

  1. 它真的能干活，不是 demo。我用它做的事情都是真的工作——拆 brief、审稿、做研究、归档信息——不是"看，它能给你写一首诗"。
  2. 它是开源的。这一点对我们做营销的人很重要。开源意味着我教它的东西、我给它的数据、我让它干的活，都在我自己的电脑或者我们公司自己的环境里，不会变成一家公司的训练数据。
  3. 它的"说明书"机制非常友好。后面会详细讲这一点——它让一个不会写代码的营销人，也能把自己的经验教给它。

B. 一分钟装好 OpenClaw

我第一次装 OpenClaw 是在 2026 年 3 月 5 号，那时候我腾空了一台 MacBook，按官方文档原生装在我的黑窗口里，整整折腾了好几个小时——要改一堆网络环境、装 Node.js、跑各种命令。

但现在不需要了。

因为这一两个月，国内几家大厂都做了封装版的 OpenClaw。我这礼拜刚装了一只飞书版的虾——

打开飞书 → 进官方插件市场 → 找 OpenClaw → 点安装 → 按提示授权 → 完成。

整个过程不到一分钟。

一分钟之后，你的飞书里就多了一个机器人，@它说话，它就开始干活。所有黑窗口、Node.js、网络配置、API key 配置——全部被飞书帮你处理掉了。

装好之后的第一句话可以是："你好，介绍一下你自己。" 它会用自己的口吻回你——这一步用来确认它已经活了。

如果你想装原生版（不推荐新手）

如果你以后想跑更复杂的活、想完全在自己的电脑上跑、不想依赖飞书，那就装原生版。装的过程现在比 3 月份简单多了，但还是会用到黑窗口。

官方安装命令一行：

  • macOS / Linux：curl -fsSL https://openclaw.ai/install.sh | bash
  • Windows PowerShell：iwr -useb https://openclaw.ai/install.ps1 | iex

装好之后跑一句 openclaw onboard --install-daemon，再跑 openclaw dashboard 就能在浏览器看到它的控制台。

新手强烈建议先用飞书版。等你觉得不够用了再装原生。

C. OpenClaw 的 Tool——它能干哪些活

OpenClaw 自带一组 Tool（中文翻译"工具"；也可以把它想象成虾的"手脚"——但官方文档和 Agent 圈里的正式叫法就是 Tool）。常用的几类：

  1. exec——执行命令。能跑黑窗口里的命令，比如下载、安装、跑脚本。
  2. read / write——读写文件。能读你电脑里的文件、能新建/修改文件。
  3. browser——打开浏览器。能上网、搜东西、读网页。
  4. code execution——跑代码。能在一个安全的小盒子里跑一段代码（比如帮你算一个表格）。
  5. 发消息——在飞书版里，它可以直接发飞书消息、贴图、回复你。
  6. ……还有更多。完整 Tool 列表见 docs.openclaw.ai/tools，但你不需要背。

OpenClaw 还有一个 permission profile（权限档位） 机制——你可以设这一只虾能用哪些 Tool。比如只让它做内容工作，就只开 read/write + browser，不开 exec。这样它就算被你教错了，也搞不出大事。

档位有四档：minimal / messaging / coding / full。前两档很安全，后两档威力大但你得清楚自己在干嘛。

D. 教养 OpenClaw——它真正变成"你的虾"的关键

这是 OpenClaw 最重要的一层，也是它和别的 AI 产品真正拉开差距的地方。

教养 = 给它写 Skill

OpenClaw 里有一个核心概念叫 Skill——你可以把它想象成"写给虾的一份说明书"，这是它在你脑子里的形象。但它的正式名字就是 Skill，下次你在 OpenClaw 文档里、在别的 Agent 产品里看到 "skill" 这个词，记住它说的就是这个东西。

一个 Skill 就是一个 markdown 文件，文件名固定叫 SKILL.md，里面长这样：

---
name: hello_world
description: 一个简单的打招呼技能。
---

# Hello World Skill

当用户让你打招呼的时候，用 echo 这个 Tool 说一句
"Hello from your custom skill!"。

就这么简单。

  • 上面那段 --- name / description --- 是给虾看的"目录"——它知道自己有这么一个 Skill、这个 Skill 是干嘛用的。
  • 下面的正文是 Skill 的内容：你想让它在什么场景下、按什么步骤、用什么语气、调用哪些 Tool——全部用大白话写下来就行。

关键的一句话：

写 SKILL.md 不是写代码，是写"工作 SOP"。

你给一个新人入职的工作交接文档，长什么样，SKILL.md 就长什么样。

这就是 OpenClaw 对营销人最友好的地方——写 SOP 是营销人的看家本领。

说明书放在哪儿——顺便把整只虾的大脑长什么样给你看

写好的 SKILL.md 放进 ~/.openclaw/workspace/skills/ 下。但与其只告诉你这一条路径，我更想把整只虾的大脑摊开给你看一眼：

~/.openclaw/workspace/
├── AGENT.md                 ← 这只虾是谁（名字、选用的模型、权限档位）
├── SOUL.md                  ← 这只虾的"灵魂"（语气、人设、行事风格）
├── USER.md                  ← 你是谁（让它认识主人——偏好、身份、工作背景）
├── MEMORY.md                ← 长期记忆（前面讲过的那个）
├── memory/
│   ├── 2026-04-10.md        ← 今天的日志
│   ├── 2026-04-09.md        ← 昨天的日志
│   └── ...                  ← 更早的日志
├── DREAMS.md                ← 做梦机制沉淀出来的东西
└── skills/
    ├── strategy/
    │   └── SKILL.md         ← 策略推演引擎说明书
    ├── content-review/
    │   └── SKILL.md         ← 审稿说明书
    └── <你新写的技能>/
        └── SKILL.md

（~ 在 Mac 上是你的用户文件夹，在 Windows 上是 C:\Users\你的用户名。）

分成三组来看：

  • 身份组（AGENT.md / SOUL.md / USER.md）——回答三件事：它是谁、它是什么性格、你是谁。这一组决定了它"像不像"你想要的那只虾。
  • 记忆组（MEMORY.md / memory/ / DREAMS.md）——回答一件事：它记得什么。长期记忆 + 每日日志 + 做梦沉淀。
  • 技能组（skills/）——回答一件事：它会干什么。你写给它的每一份 SOP。

一只虾的全部大脑，就是这么一个文件夹。没有神秘的数据库，没有云端黑盒，没有 SaaS 后台——全部是看得懂、改得动、备份得走的 markdown 文件。

这张图是整场分享里最重要的一张：

身份组告诉它"自己是谁、你是谁"；记忆组让它"记得你们一起做过什么"；技能组是你教它的 SOP。

这几个文件加起来，就是一只你自己养出来的数字同事的全部。

新写好的 SKILL.md 放进 skills/<你的技能名>/SKILL.md 之后，跟你的虾说一句"重新加载一下技能"，或者重启它一下，它就读到了。之后只要你触发了这个 skill 描述里说的场景，它就自动按你说明书里的步骤干活。你不用每次都把 SOP 复制粘贴给它。

Tool / Skill / Plugin 的关系（一句话讲清楚）

这三个词新手最容易搞混。一句话理清它们的层级关系：

Plugin
打包礼盒 — 把几个 Skill 打包送同事
Skill
你教它的 SOP — SKILL.md
Tool
它生下来就有的手脚
  • Tool：具体的某个动作能力，比如"打开浏览器"。OpenClaw 自带的，是它的"手脚"。
  • Skill：你写给虾的一份 SOP，告诉它在什么场景下、怎么用这些 Tool。是你"教"它的"说明书"。
  • Plugin：把几个相关的 Skill 打包成一个安装包，方便分享给别人。是教完之后的"打包礼盒"。

你可以这么记：

Tool 是它生下来就有的；Skill 是你教它的；Plugin 是你教完之后打包送给同事的。

D-2. 真实案例：我自己教给虾的两个 Skill

两个我真的在用的 Skill。

我的虾（也就是 ICC 那只服务公司的策略 + 审稿 BoT）目前长着两个核心 Skill：

Skill 一：策略推演引擎

这是一份我自己写的 SOP，教虾怎么帮我们拆一份品牌 brief 并产出策略推演。

里面写了什么？

  • 拿到一个 brief，先按什么维度拆问题
  • 必须产出一个 KOL × Campaign / KOL × Always-On / KOX × Campaign / KOX × Always-On 的四象限矩阵
  • 每个象限里要给出几个候选方向
  • 每个方向要附上"为什么这个方向成立"的判断逻辑
  • 哪些情况下这个方向不成立（边界条件）

这一份 SOP，原本只存在于我和 ICC 几个核心同事的脑子里。写下来交给虾之后，任何一个新同事来的时候，向虾发起一个"帮我拆这个 brief"，就能拿到一份接近我手工产出的草稿。

她不是在替代我，她是在让我和我的同事们的判断变得可复用。

Skill 二：审稿（Content Review）

这是我教虾审稿的 SOP。审一篇要发出去的内容稿，从四个维度过：

  1. 产品保真：产品信息、参数、机制、卖点有没有写错、有没有夸大
  2. Brief 合规：是不是按 brief 的核心信息和优先级写的，关键信息有没有漏
  3. KOL 风格适配：如果是给某个 KOL 写的，语气、用词、节奏、人设有没有跑偏
  4. 品牌红线：违反品牌红线的句子（每个客户的红线列表都不一样，写在 skill 里）

它过完之后会按这四个维度逐条挑问题，附原句和修改建议。

这一份 skill 也被我推到了一个私有 GitHub 仓库里——这样我在任何一台装了 OpenClaw 的电脑上，都能把它拉下来给我的虾用，公司同事也能拉。

关键：这两个 skill 都是私有的，没有公开。你的经验是你的资产，用 OpenClaw 教养它的过程，就是把你脑子里的资产沉淀成一份可复用的、归你自己所有的"数字员工说明书"。

这跟把数据交给某个 SaaS 厂商训练它们的模型，完全是两回事。

E. 一个公司怎么部署多只虾（多 Agent）

一个公司里可能不止一个人需要虾，也不止一个场景需要虾。

OpenClaw 在飞书版上的玩法是这样的：

一个飞书企业可以装多只虾。每只虾是一个独立的 OpenClaw 账号，每个账号绑定一组不同的说明书、不同的人设、不同的脑子（大模型）、不同的工作区。

打个比方：

虾的名字 给谁用 装了哪些说明书 用的脑子
策略虾 策略组同事 策略推演引擎 Claude（更擅长长链推理）
审稿虾 内容组同事 审稿 SOP + 各品牌红线 DeepSeek（中文语感好、便宜）
（未来可扩展） 研究 / 客户管理 / 运营报表 按需要再写 按需要选脑子

每只虾在飞书里就是一个机器人，@它就是在跟它说话。一个公司可以养出十几只各管一摊的虾。

Vibe Coding 入门

不写代码，用嘴搭网站——以这个站为例

💜 Wendy 私心分享一遍我的 case 给你看——对未知的恐惧，「看过一遍」就解啦~

起点：为什么有这个站

三个月前，我跟你现在一样。

最近很多朋友来找我聊 AI。他们的困境跟我三个月前一模一样——市面上 AI 教程不是没有，是实在太多了。多到眼花缭乱，反而不知道从何下手。很多人是被小龙虾（OpenClaw）吸引过来的，但真要上手的时候，又不知道先做什么。

所以我就想：把这几个月积累的东西汇总起来，做成一个集合地，给我的朋友们一个"从这里开始"的入口。这个网站的名字就叫 ai-intro-for-friends——给朋友的 AI 入门。

你现在看的这整个 Playbook，就是我用嘴"写"出来的。

边干边学，是唯一靠谱的路径

这几个月自己摸索下来，我最大的体感是：

既不能纸上谈兵只看教程不动手——看十篇不如做一次。

也不能闭眼先动手——人对未知的产品，总是有恐惧的。

边干边学，是唯一靠谱的路径。

Vibe Coding 恰好是"边干边学"的最佳实践——反馈循环极短。你说一句话，AI 写一段，你马上就能看到结果；不对就改，改了再看。整个过程你不是在"读教程"，你是在"做东西"——而你脑子里的知识，是在做的过程中自己长出来的。

什么是 Vibe Coding

Vibe Coding 是最近一两年冒出来的词，直译"氛围编程"。意思是：

你不写代码。你告诉 AI 你想要什么，AI 帮你写。

你的活是三件：

  • 把需求说清楚——"我想要一个什么样的东西"
  • 把 AI 的结果看明白——哪里对、哪里不对、哪里可以再好一点
  • 不满意就告诉它哪里不满意——"字太小""颜色不对""加个目录"

这件事最早只在程序员圈里流行，因为他们看得懂代码、知道怎么指挥 AI。但这一两年工具越来越好用，非技术背景的人也能做——只要你会说话。

从 0 搭一个 website，要经过哪几个工序

你可能会以为"做网站 = 写很多代码"。但这件事拆开来看，其实就是三个工序，每个工序对应一个工具：

1
工序一
想清楚 + 让 AI 写
工具：
Claude Code 或 Cursor
→
2
工序二
存起来
（代码有个家）
工具：
GitHub（代码仓库）
→
3
工序三
让别人能打开
（上线）
工具：
Render / Vercel / GitHub Pages

三个工序是接力关系——上一棒做完，交给下一棒：

  • 工序一：你跟 AI 对话，在你自己的电脑上写出一份能跑的代码。但这时候它只在你电脑里，别人看不到。
  • 工序二：把代码推到 GitHub，相当于放进一个"作品仓库"。这一步解决的是"代码得有个地方存着"——不然你换台电脑就丢了，也没法分享给别人。
  • 工序三：从 GitHub 里把代码拉出来，部署到一个能被访问的公网地址上。这一步之后，你有了一个 xxx.onrender.com 这样的链接，谁都能打开。

就这三步。没有了。

即使你今天不打算搭网站，知道"原来做一个网站就这三步"，以后再听人聊什么前端后端部署 CI/CD，也不会再懵。

实战：我是怎么走完这三个工序的

先说结论：第一版从打开 Claude Code 到网站上线，前后不到 1 小时。

下面是完整过程。

Step 1：打开 Claude Code，把想法 summary 给它

我打开 Claude Code，把脑子里的想法跟它说了一遍：

我想做一个网页，给我的营销朋友们看。分享三件事：

1. 基本的 AI 概念（AI 世界地图）
2. 我自己用下来的几个 tips
3. 从 0 上手小龙虾（OpenClaw 安装到 skill 定制）

就这样。没有任何代码，没有任何技术词——就像跟一个懂技术的朋友聊"我想做一个什么样的东西"。

它会自己决定用什么技术栈（这一步你完全不用管），帮你创建文件、生成初版内容。

Step 2：它写，你看，不好看就说

第一版写完之后，老实讲——不太好看。

我就告诉它："改成 Notion 风格。" 它就重做了一版视觉。

这个循环整个过程里不断重复：

  • 字太小 → "正文字号大一点"
  • 想加目录 → "左边加个导航"
  • 颜色不对 → "主色换成紫色"
  • 段落太挤 → "行距松一点"

不用想术语，用大白话描述你想要的。它会自己翻译成代码。

工序一到这里就做完了——你电脑里有了一份能在浏览器里打开的 HTML。

Step 3：推到 GitHub（工序二）

GitHub 是什么？简单讲，就是你的作品仓库。代码存在那里，你自己能从任何一台电脑拉下来接着改，也能分享给别人。

怎么推？我也不会——就问 Claude："现在怎么把这个推到 GitHub 上？"

它一步一步告诉我怎么做：注册账号、建仓库、装 Git、把本地代码推上去。每一步我只要照着做。遇到卡点，截图发给 Claude，问"这里我该怎么办"，它就告诉我下一步。

你只需要会复制粘贴 + 会截图。

Step 4：部署到 Render（工序三）

推到 GitHub 之后，网站还没上线——没人能打开。这一步要把它部署到一个能被公网访问的地方。

我选的是 Render（免费、简单、对新手友好）。流程一样——不会就问 Claude，卡了就截图。它会告诉你：

  • 去 Render 注册账号
  • 把你的 GitHub 仓库授权给它
  • 点几下按钮，配置一下
  • 等它自己跑完

Step 5：它活了

Render 会给你一个链接，比如 xxx.onrender.com。点进去——就是你刚刚"说"出来的网站。

这一刻非常魔法。你刚才还在一个聊天框里描述你想要什么，现在它真的在互联网上存在。你可以把这个链接发给任何人，任何人都能打开。

从打开 Claude Code 开始，到这一刻，不到 1 小时。

走完一遍之后的那个感觉

老实讲，我没什么特别踩坑的。第一次走完之后，最强烈的感觉是：

"Wow！原来如此——原来轻量级部署一个 website 就这几个步骤。"

当你真的走过一遍，对"网站是怎么做出来的""代码是怎么上线的"这些原本神秘的东西，就完全没有了陌生的恐惧。

还有一件特别重要的事——因为全程有 Claude 在：

我完全不会害怕或羞耻问出一些"蠢问题"。

它不会笑话我，它只会一步一步教我。

这件事让整个过程推进得特别顺利。

这点再怎么强调都不过分。非技术背景的人最大的心理障碍，不是"我学不会"，而是"我怕问出一个让人觉得我很蠢的问题"。跟 Claude 协作的时候，这个心理障碍直接没了——你问多蠢的问题它都会认真回答。

然后你就会想迭代

有了第一次的信心，你就会想继续折腾它。

你现在看到的这个多 tab 版本，就是我最近的改版——从一个单页的长文，改成了一个有首页、有板块、会继续生长的站。这次改版也是 vibe coding 做的。

但我得诚实说一件事：

一旦页面变复杂，设计和 UX / UI 的规划就比内容本身难多了。

哈哈，这个一直是我欠缺的部分。我在和 AI 一起学习、一起进步。

Vibe Coding 不是魔法。它让启动变得很轻——让你有机会从"完全不敢开始"变成"做出了第一版再说"。但往后的精细度、设计感、专业度——依然要你自己持续学、持续迭代。

对我来说这挺好的——它让我有了动手的勇气，也让我看见自己还缺什么。这本身就是成长。

你也可以开始

随便想一个你自己的小东西：

  • 一份给客户的提案，想做成可分享的网页
  • 一个你自己的简历 / 作品集
  • 一个整理 KOL 资料的小工具
  • 一个给朋友的 xxx——就像我给朋友的 AI 入门

不一定要是"网站"。Vibe Coding 可以做网页、做小程序、做自动化脚本、做数据表格。本质都是：把你脑子里模糊的想法，用嘴讲给 AI 听，让它帮你变出来。

启动成本是零。你唯一要跨过去的，是"我要不要开始"那一步。

你不需要变成程序员。你只需要开始做一件你以前觉得自己做不了的事。

好用的 AI 工具

不做全景测评，只讲我真的在用的

开场：只推跑通过的

市面上"AI 工具推荐"的文章铺天盖地，大多数看完你还是不知道该从哪个开始。

我想给你的是另一种——我只推我真的在用、而且用了不是一两天的。每个产品都讲清楚三件事：

  • 我拿它干嘛（具体活儿，不是功能清单）
  • 它最让我离不开的点
  • 我踩过的坑 / 不爽的地方

最后给你几个跨产品组合拳——单独用一个工具是一回事，搭起来用是另一回事。

⚠️ 先说一件事

对于中国区伙伴来说，下面这些 AI 工具几乎都需要"科学上网"——Claude Code、Cursor、Gemini、NotebookLM 基本全都绕不过去。所以在折腾这些 AI 之前，你得先把网络这件事搞定，不然连登录都进不去。

这也是为什么我把 Nexitally 单独拎出来，放在所有 AI 工具前面讲——它不是 AI，但它是让这一切能跑起来的前提。

先把地基打好

这个小节只有一个工具，但它是前置中的前置——跳过它，后面 6 个 AI 工具大概率一个都用不成。

🌐 Nexitally
让上面那些 AI 工具能跑起来的"地基"
我拿它干嘛
它是我日常访问 Claude Code、Cursor、Gemini、NotebookLM、GitHub 等服务的网络通道。没有它，后面那 6 个工具我基本都打不开。
离不开的点
算是我用过的比较稳定的一个"科学"工具。当然，你手里如果已经有趁手、稳定的，继续用就好——这里重点是"稳定"本身，不是非得用哪一家。

6 个 AI 工具

按使用频次和依赖度从高到低排。越往上的，我每天都开；越往下的，有特定场景才会想起来。

🤖 Claude Code
我 70% 的活都是用它干的
我拿它干嘛
写网页、搭知识库、做 AI 营销变局的沙盘、和我一起写 skill、甚至公众号的文章和配图——都是 Claude Code 做的。
离不开的点
一个工具把脑力活、工程活、创作活都接住了。几乎没有我想做、但它干不了的事。说它是我的"主操作台"也不为过。
踩过的坑
对网络环境和账号要求太严格了，对简中区用户非常不友好。建议先把网络环境搞定再开始用，不然一上来就会被劝退。
📝 Cursor
Claude Code 的背靠背搭档
我拿它干嘛
能做的事几乎和 Claude Code 一样，是我的 Plan B。尤其是本地文件相关的活，我会优先丢给 Cursor。
离不开的点
国内支付友好、网络环境要求相对宽松、还能切最好的模型（Opus 4.6、GPT 5.4 都有）。对简中区用户来说上手门槛低得多。
小建议
两个都用的话，容易忘记自己刚才在哪个里跟 AI 聊过什么。可以按场景分开——比如本地文件的活交给 Cursor，其他的给 Claude Code。
🎨 Genspark
非设计师的 PPT 和出图救星
我拿它干嘛
agent 功能里的 PPT 生成 和 nanobanana 出图。这两个是我高频使用的。
离不开的点
它把 PPT 和 nanobanana 封装过一次——对不是设计师的我非常友好。直接出结果，不用自己调参数、不用懂设计原理。
适用场景
适合"我要一个差不多的东西，赶紧上"。如果你想精细控制每一个细节，它不够用——但多数时候我们不需要那么精细。
💎 Gemini
有灵性的脑暴搭子
我拿它干嘛
deepresearch（深度调研）+ brainstorming（创意脑暴）。发散思考那一端的活，我交给它。
离不开的点
是我用过的模型里，感觉"比较有灵性"的一个。你抛一个模糊的想法过去，它能接住，再反弹回来一些你没想到的角度。
小建议
它是"发散"那一端。收敛、整理的活交给 NotebookLM 更合适。两个搭着用才是完整的闭环。
📚 NotebookLM
读不过来的材料，让它帮你可视化
我拿它干嘛
把一堆文本材料（PDF、文章、笔记）喂进去，让它可视化地整理成卡片 / PPT / 视频 / 播客。
离不开的点
文本读不过来的时候，用可视化来协助理解——大脑接受起来轻松很多。尤其是它的播客功能，通勤路上可以"听"掉一份研究报告。
小建议
它是"整理"工具不是"创造"工具，别指望它替你思考。材料的选择和判断还是你的活。
📦 GitHub
从"skill 仓库"到"我自己的备份"
我的理解演变
一开始我以为它就是个 skill 仓库（经常会刷到什么爆火的 skill，从那里下载）。后来自己做网页了才发现——它是很多代码的仓库，也可以是我所有项目的备份。
离不开的点
一旦你开始自己做东西，GitHub 就会从"别人的货架"变成"你自己的家"。你的每一版改动它都记得。
踩过的坑
界面对新手确实有点劝退，但你不需要全懂——让 Claude Code 或 Cursor 帮你 commit 和 push 就够了。你只要知道"代码去哪了"就行。

组合拳：搭起来用，1+1 > 2

单独介绍每个工具只是第一步。真正能跑起来的工作流，都是几个工具搭配着用的——一个负责 A，一个负责 B，接力完成一件事。

🔍 搜资料 + 消化知识
一个负责"想到"，一个负责"看清"
Gemini → NotebookLM

想研究一个新话题？先让 Gemini 做一轮 deepresearch，把散落在各处的信息抓一遍；然后把结果喂给 NotebookLM，让它整理成卡片、PPT 或者播客。

Gemini 在"发散 & 找到"这一端特别强，NotebookLM 在"收敛 & 看清"那一端特别强。两个搭着用，一篇复杂的报告可以在一两个小时里就变成你大脑能吸收的形状。

🎨 做 PPT / 出图
对非设计师来说性价比最高的单品
Genspark （一个就够）

这个"组合"其实只有一个工具——但我还是想单独拎出来讲，因为它替代的是"找设计外包"这件事。

你不需要自己会 Figma，不需要会 Photoshop，不需要学 PPT 模板审美。把需求告诉 Genspark 的 agent，几分钟出结果，大部分场景下够用。对非设计背景的人来说，这是我觉得最值得付费的工具之一。

🌐 搭一个网站（工具视角）
三个工具，各自扮演一个角色
Claude Code → GitHub → Render

Claude Code 负责写——你用嘴告诉它要什么，它把代码写出来、在本地能看到效果。

GitHub 负责存——代码要有个家，它是你所有改动的历史档案馆。

Render 负责上线——把 GitHub 上的代码变成一个别人能打开的网址。

你就是这三个工具的"指挥"，每个工具接一段活，串起来就是一个跑起来的网站。

这里讲的是工具视角（每个工具干嘛）。如果你想看完整工序（从 0 到 1 的全流程）——去 Vibe Coding tab，那边是实战版。

写在最后：工具会换，原则不会

这份清单是 2026 年 4 月的版本。过几个月可能就不对了——新的工具会冒出来，旧的工具会被替代。

但有三条原则不会变：

  • 只推跑通过的——没真的用过的工具，再火我也不推荐给你
  • 跟着活选工具，不是反过来——先想清楚你要做什么，再找对应的工具，不要看到新工具就想"我能拿它干嘛"
  • 边用边换，不求长情——哪个顺手用哪个，这个月用 A，下个月发现 B 更好就换 B，工具是你的，不是你伺候它

最好的工具，是那个能让你今天就开始动手的。
